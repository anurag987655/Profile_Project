# Building a Compiler in 2026

Last winter I set out to write a compiler for *Cypher* — a small systems language with manual memory management and algebraic data types. The journey from text to executable taught me more about computer architecture than any textbook.

## Lexing & Parsing

I used a hand-written lexer (faster and simpler than flex for a small grammar) and a recursive-descent parser. Recursive descent is wonderfully straightforward: one function per production rule. Error messages, however, are brutal — you need to track what tokens were expected at each position, not just where the parser choked.

## IR and Optimisation

The intermediate representation was a simple three-address code (TAC). Constant folding, dead code elimination, and copy propagation were surprisingly easy to implement and gave roughly 15% speedup on average. The real win came from register allocation: a linear-scan allocator that maps virtual registers to the 16 x86-64 GPRs.

## Codegen

Generating x86-64 assembly by emitting strings is oddly satisfying. The trickiest part was the calling convention — saving callee-saved registers, aligning the stack, and passing arguments in the right order (`RDI`, `RSI`, `RDX`, `RCX`, `R8`, `R9`). Seeing "Hello, world!" printed from self-generated machine code is a rite of passage every programmer should experience.

---

### Lessons Learned

1. **Hand-written lexers** are simpler than lexer generators for small languages.
2. **Three-address code** is a surprisingly effective IR for optimisation passes.
3. **Linear-scan register allocation** gives 90% of the benefit with 10% of the complexity of graph colouring.
